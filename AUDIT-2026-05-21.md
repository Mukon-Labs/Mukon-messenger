## Executive Summary

Mukon Messenger is a **privacy-first wallet-to-wallet messenger** with solid cryptographic foundations (NaCl E2E encryption, on-chain key backup, Arcium MPC v0.8.0). The architecture is innovative and the crypto is sound. However, **critical backend security gaps** exist that could enable DoS attacks and abuse before mainnet launch.

### Top 3 Critical Issues
1. **🔴 CRITICAL: No rate limiting on backend** — WebSocket and REST endpoints completely unprotected (DoS risk)
2. **🔴 CRITICAL: Missing input validation** — Backend trusts all client data without size/format checks
3. **🟡 HIGH: Session key security** — No key rotation policy, no revocation audit trail, permissive validity periods

### Positive Highlights
- ✅ Strong E2E encryption (NaCl box/secretbox)
- ✅ Proper canonical ordering for relationship PDAs
- ✅ Session keys eliminate wallet popup spam
- ✅ On-chain group key backup (unique feature vs WhatsApp/Signal)
- ✅ Local-first message architecture
- ✅ Arcium MPC integration working on devnet

## 🔴 Critical Security Issues

### 1. Backend: Zero Rate Limiting
**Location:** `backend/src/index.js`  
**Impact:** HIGH — DoS attack vector, spam messages, resource exhaustion  
**Effort to fix:** Small (1-2 hours)

**Issue:**
- No rate limiting on REST endpoints (`/messages`, `/group-messages`, `/health`)
- No rate limiting on Socket.IO events (`send_message`, `send_group_message`, `share_group_key`, etc.)
- Attacker can:
  - Spam backend with unlimited message POST requests
  - Flood WebSocket rooms with encrypted garbage
  - Exhaust Postgres connections
  - Fill disk with pending messages (pre-acknowledgement)

**Evidence:**
```bash
$ grep -r "rate.limit\|rateLimit\|throttle" backend/
# No results
```

**Recommendation:**
Implement `express-rate-limit` for REST and custom per-socket rate limiting:
```javascript
// REST
const rateLimit = require('express-rate-limit');
const limiter = rateLimit({
  windowMs: 60 * 1000,    // 1 minute
  max: 100,               // 100 requests per minute per IP
  message: 'Too many requests',
});
app.use('/messages', limiter);

// Socket.IO (custom)
const socketRateLimits = new Map(); // socket.id -> { count, resetAt }
function checkSocketRateLimit(socketId, maxPerMinute = 60) {
  const now = Date.now();
  const limit = socketRateLimits.get(socketId) || { count: 0, resetAt: now + 60000 };
  if (now > limit.resetAt) {
    limit.count = 0;
    limit.resetAt = now + 60000;
  }
  limit.count++;
  socketRateLimits.set(socketId, limit);
  return limit.count <= maxPerMinute;
}
```

**Priority:** 🔴 **Fix before mainnet**

### 2. Backend: Missing Input Validation
**Location:** `backend/src/index.js`, all Socket.IO handlers  
**Impact:** MEDIUM-HIGH — Injection risks, buffer overflows, malformed data crashes  
**Effort to fix:** Medium (3-4 hours)

**Issue:**
Backend trusts all client inputs without validation:
- No size limits on `encrypted`, `nonce`, `content` fields
- No format validation on pubkeys (could send non-base58)
- No length checks on `conversationId`, `groupId` (could be empty or 10MB string)
- No type checks (could send arrays instead of strings)

**Evidence:**
```javascript
// backend/src/index.js:238 — no validation
const { conversationId, encrypted, nonce, sender, signature } = req.body;
// Immediately used without checks!
```

**Attack vectors:**
- Send 100MB `encrypted` string → exhaust memory/DB storage
- Send empty `conversationId` → crashes on DB insert (foreign key violation)
- Send non-base58 `sender` → crashes `verifySignature` with bs58 decode error
- Send array instead of string for `nonce` → crashes `Buffer.from(nonce, 'base64')`

**Recommendation:**
Use `joi` or `zod` for schema validation on all endpoints and socket events:
```javascript
const Joi = require('joi');

const messageSchema = Joi.object({
  conversationId: Joi.string().min(10).max(200).required(),
  encrypted: Joi.string().max(100000).required(), // ~100KB limit
  nonce: Joi.string().base64().max(100).required(),
  sender: Joi.string().regex(/^[1-9A-HJ-NP-Za-km-z]{32,44}$/).required(), // base58 pubkey
  signature: Joi.string().base64().max(200).required(),
  replyTo: Joi.string().max(100).optional(),
});

// In handler:
const { error, value } = messageSchema.validate(req.body);
if (error) return res.status(400).json({ error: error.details[0].message });
```

**Priority:** 🔴 **Fix before mainnet**

### 3. Session Key Security Gaps
**Location:** `programs/mukon-messenger/src/lib.rs`, `app/src/contexts/MessengerContext.tsx`  
**Impact:** MEDIUM — Stale session keys, no rotation, permissive validity  
**Effort to fix:** Medium (2-3 hours)

**Issues:**
1. **No automatic key rotation** — Client creates session with `valid_until = now + 30 days` and never rotates
2. **No revocation audit trail** — `revoke_session` exists but app never calls it except on disconnect
3. **Permissive validity check** — Program checks `valid_until` but client can set arbitrarily long periods
4. **Low balance recovery risky** — If session key runs out of SOL, client silently falls back to wallet signing (confusing UX)

**Evidence:**
```rust
// lib.rs:169 — No maximum validity constraint
pub fn create_session(ctx: Context<CreateSession>, valid_until: i64) -> Result<()> {
  // Client can pass valid_until = year 3000!
}
```

```typescript
// MessengerContext.tsx:557 — Hardcoded 30-day validity, never rotated
const validUntil = Math.floor(Date.now() / 1000) + 30 * 24 * 60 * 60; // 30 days
```

**Recommendation:**
1. **Program-side max validity:** Enforce `valid_until <= now + 7 days` in `create_session`
2. **Client-side rotation:** Auto-rotate session keys after 3 days
3. **Revocation on logout:** Call `revoke_session` when user disconnects wallet
4. **Better balance monitoring:** Warn user when session key balance < 0.005 SOL

**Priority:** 🟡 **Fix before scale** (not blocking for hackathon demo)

**Location:** Multiple files — `MessengerContext.tsx`, backend socket handlers, `ChatScreen.tsx`  
**Impact:** MEDIUM — Poor UX, debugging nightmares, silent data loss  
**Effort to fix:** Medium (4-5 hours across codebase)

**Examples:**

  // User never sees this!

**Priority:** 🟡 **Medium** (affects UX, not security)

### 5. Race Conditions: Group Key Distribution
**Location:** `MessengerContext.tsx` (group key handling), backend `share_group_key` event  
**Impact:** MEDIUM — Members can't decrypt messages if they miss key distribution  
**Effort to fix:** Small (already has fallback, needs refinement)

**Issue:**
When admin invites multiple members simultaneously:
1. Admin calls `store_group_key_for_member` on-chain for each invitee ✅
2. Admin also emits `share_group_key` socket event for immediate delivery ✅
3. If invitee is offline during both, they can `request_group_key` ✅

**BUT:** If invitee accepts invite BEFORE on-chain key is indexed (Solana takes 400-800ms):
- Client calls `acceptGroupInvite`
- Immediately tries to fetch `GroupKeyShare` PDA
- Account doesn't exist yet (transaction not finalized)
- Client emits `request_group_key` to socket
- Other members might not be online → stuck

**Evidence:**
```typescript
// MessengerContext.tsx:1890 — No retry logic if account not found
const keyShareAccount = await connection.getAccountInfo(keySharePDA);
if (!keyShareAccount) {
  // Tries socket fallback once, but what if no one online?
}
```

**Recommendation:**
Add retry logic with exponential backoff:
```typescript
const fetchGroupKeyWithRetry = async (pda: PublicKey, maxRetries = 5) => {
  for (let i = 0; i < maxRetries; i++) {
    const account = await connection.getAccountInfo(pda);
    if (account) return account;
    await new Promise(r => setTimeout(r, 2 ** i * 500)); // 500ms, 1s, 2s, 4s, 8s
  }
  throw new Error('GroupKeyShare not found after retries');
};
```

**Priority:** 🟡 **Medium** (rare edge case, but frustrating when it happens)

### 6. Nonce Reuse: Potential Replay Issues
**Location:** `app/src/utils/encryption.ts`, group message encryption  
**Impact:** LOW-MEDIUM — Nonce collisions could leak plaintext relationships  
**Effort to fix:** Small (1 hour)

**Issue:**
- DMs use `nacl.randomBytes(24)` for nonce ✅ (secure)
- Group messages ALSO use `nacl.randomBytes(24)` ✅
- **BUT:** Backend de-dupes by `(encrypted, nonce, sender)` tuple
  - If same plaintext sent twice with different nonces → both stored
  - If attacker replays old `(encrypted, nonce, sender)` → backend rejects
  - **However:** Nonce collision probability is ~2^-96 (astronomically low)

**Current de-dupe logic:**
```typescript
// MessengerContext.tsx:814
const isDuplicate = currentMsgs.some(
  (m) => m.encrypted === encrypted && m.nonce === nonce && m.sender === sender
);
```

**Risk assessment:**
- Nonce collision: Not a realistic threat (need 2^48 messages for 50% collision)
- Replay attack: Already prevented by de-dupe logic ✅
- **Actual issue:** De-dupe uses *client-side state* only
  - If client clears cache, same message could be added again
  - Solution: Backend should also track seen `(encrypted+nonce+sender)` hashes

**Recommendation:**
Backend-side de-dupe before saving to Postgres:
```javascript
// In saveMessage / saveGroupMessage
const messageHash = crypto.createHash('sha256')
  .update(encrypted + nonce + sender)
  .digest('hex');

await db.query(
  'INSERT INTO messages (..., hash) VALUES (..., $X) ON CONFLICT (hash) DO NOTHING',
  [..., messageHash]
);
```

**Priority:** 🟢 **Low** (defense-in-depth, not urgent)
